# Technical Reference

## Main Surfaces

- `source/BodyMetricsDomain.mc`: compatibility facade consumed by the UI.
- `source/usecases/*`: application workflows.
- `source/renderers/*`: pure rendering.
- `source/policies/*`: classification and threshold rules.
- `source/i18n/*`: locale catalog and lookup.

## Persistence

### Key Components

- `source/BodyMetricsDataProvider.mc` manages measurements and data source tracking.
- `source/BodyMetricsHistory.mc` manages history and debug history.
- `source/BodyMetricsTargets.mc` manages user targets and effective targets.
- `source/BodyMetricsGarminProfile.mc` adapts the Garmin profile into an application format.

### Rules

- History is the authoritative source for trend data.
- Trend cache must be invalidated after save, reset, and relevant mutations.
- Storage details must stay aligned with code before being promoted to external documentation.

## Calculations and Policies

- BMI, BMR, muscle percent, and power are derived through application formulas.
- Classification uses dedicated policies and thresholds generated by the threshold factory.
- Effective targets depend on user target, default fallback, and metric policy.

## UI Contracts

- The view coordinates modes and state but must not talk to storage directly.
- Renderers receive prepared models and do not perform side effects.
- Input and menu delegates translate events into state transitions.
