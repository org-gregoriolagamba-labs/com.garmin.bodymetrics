# Technical Reference

## Main Surfaces

- `source/BodyMetricsDomain.mc`: compatibility facade consumed by the UI.
- `source/usecases/*`: application workflows.
- `source/renderers/*`: pure rendering.
- `source/policies/*`: classification and threshold rules.
- `source/i18n/*`: locale catalog and lookup.
- `source/renderers/RendererCommon.mc`: global functions shared across all renderers.

## Persistence

### Key Components

- `source/BodyMetricsDataProvider.mc` manages measurements and data source tracking.
- `source/BodyMetricsHistory.mc` manages history and debug history, with an in-memory cache.
- `source/BodyMetricsTargets.mc` manages user targets and effective targets.
- `source/BodyMetricsGarminProfile.mc` adapts the Garmin profile into an application format.

### Rules

- History is the authoritative source for trend data.
- Trend cache must be invalidated after save, reset, and relevant mutations.
- The Domain must not redefine storage key constants: the authoritative copies live in the respective use cases.

## Calculations and Policies

- BMI, BMR, muscle percent, and power are derived through application formulas.
- Classification uses dedicated policies and thresholds generated by the threshold factory.
- Effective targets depend on user target, default fallback, and metric policy.
- `potenzaRange()` is derived directly from `muscleKgRange()` by scaling each threshold by 35: no duplicated constants.
- `round1Global()` and `fmt1Global()` are the only authoritative implementations of one-decimal rounding and formatting.

## UI Contracts

- The view coordinates modes and state but must not talk to storage directly.
- Renderers receive prepared models and do not perform side effects.
- Input and menu delegates translate events into state transitions.
- The MENU key is intercepted by `onMenu()` in the InputDelegate: in wizard mode the event is consumed without opening the system menu.

## System Info Screen

- Accessible from menu `System` -> `Information`.
- Displays: app name, version, release date, author, website.
- The "Website" entry in the system info screen is an actionable button (`:action => true`, `:actionId => :open_qrcode`): pressing ENTER opens `BodyMetricsQrcodeView`.
- `BodyMetricsQrcodeView` loads `Rez.Drawables.QrcodeWebsite` (120×120 px PNG) and draws it centered on a full-screen black background.
- The QR code resource is declared in `resources/drawables/drawables.xml`; the file is `resources/drawables/qrcode_website.png`.

### Build Parameters (VS Code Task)

The task `Monkey C: Build and Run BodyMetrics (fr265)` in `.vscode/tasks.json` invokes `.vscode/run-bodymetrics-sim.sh`.
The device target is configurable in the script; full/lite mode will be selectable via an environment variable or script argument once the lite variant is implemented.

### Lite Build (planned)

The lite build will use a dedicated jungle file (`monkey-lite.jungle`) and separate sources that exclude the IT, FR, and ES localizations. The primary targets are FR55 and FR735XT. No user-facing features will be removed.
